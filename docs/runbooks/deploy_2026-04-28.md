# Hetzner deploy — 2026-04-28 sweep + post-sweep follow-ups

This runbook walks through deploying the 54 commits between
`1e70eda` (last shipped tag) and the current `main` (after the
2026-04-28 gap-closure PR merges) onto the Hetzner production box.

The deploy is **non-destructive**: no DB migrations, no breaking config
changes, no API contract changes. Rollback is `git checkout 1e70eda &&
docker compose build && up -d`.

## Pre-deploy checks

```bash
# On laptop, confirm the branch you're about to deploy.
git fetch origin
git log --oneline origin/main..HEAD            # should be empty after merge
git rev-parse HEAD                             # capture for rollback log

# Confirm green CI on the merge commit before proceeding.
gh run list --branch main --limit 5
```

## Deploy

SSH to the Hetzner box and:

```bash
cd /path/to/hc-stark
git fetch origin
PREV_REV=$(git rev-parse HEAD); echo "Rolling forward from $PREV_REV"
git checkout main
git pull --ff-only

# Rebuild and roll. Compose recreates only changed images.
docker compose build hc-server hc-mcp-http
docker compose up -d hc-server hc-mcp-http

# Watch the boot for ~60s. The hc-worker validation runs at startup;
# if HC_SERVER_WORKER_PATH points at something missing or non-executable,
# hc-server refuses to come up. This is intentional.
docker compose logs -f --tail=200 hc-server | head -100
```

## Post-deploy verification

Within ~5 min of deploy, all of these must hold:

```bash
# 1. Health
curl -fsS https://api.tinyzkp.com/healthz
curl -fsS https://api.tinyzkp.com/readyz

# 2. Templates discovery (no auth)
curl -fsS https://api.tinyzkp.com/templates | jq 'length'   # → 6

# 3. End-to-end prove + verify with a free-tier key
TINYZKP_API_KEY=tzk_test_...  # any free-tier key on the box
curl -fsS -X POST https://api.tinyzkp.com/prove/template/range_proof \
  -H "Authorization: Bearer $TINYZKP_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"params":{"min":0,"max":100,"witness_steps":[42,44]}}' | jq

# Then poll /prove/<job_id> until status: completed and /verify the proof.

# 4. MCP transport (anonymous lane bounded by global concurrency cap)
curl -fsS -X POST https://mcp.tinyzkp.com/mcp \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/list"}' | jq '.result.tools | length'  # → 10

# 5. Prometheus metrics endpoint exposes the new histograms
curl -fsS https://api.tinyzkp.com/metrics | grep -E 'sqlite_lock_wait|worker_spawn|queue_depth'
```

## Watch for the first 24h

- **Stripe MeterEvent freshness**: hourly cron runs
  `billing/sync_usage.py`. Watch for the `unbilled rows aged > N hours`
  alert in Alertmanager. The freshness check warns if rows age past
  Stripe's idempotency dedup window.

### Stripe webhook handler runs on systemd, not docker compose

`provision_tenant.py` (Stripe webhook + magic-link + contact-form
handler) runs as systemd unit `hc-billing-webhook.service` bound to
`127.0.0.1:5001`. Caddy reverse-proxies `webhook.tinyzkp.com` to it.
**Do not look for it in `docker compose ps`** — the prior orphan
container was removed 2026-04-29. To deploy code changes to
`billing/provision_tenant.py`:
```
ssh root@46.225.78.136
cd /opt/hc-stark && git pull --ff-only
systemctl restart hc-billing-webhook
```
A 2-second restart is enough for gunicorn's 2 workers to fork with
the new code.
- **Worker-spawn cap saturation**: Grafana panel "permits available"
  should not pin at zero for more than transient spikes. If it does,
  bump `HC_SERVER_MAX_WORKER_SPAWN` (default 32).
- **MCP per-plan RPM rejections**: 429s on authenticated MCP lane
  should be cleanly attributable to one tenant; if the global
  anonymous cap (`HC_MCP_MAX_INFLIGHT`) is saturating, consider
  flipping `HC_MCP_REQUIRE_AUTH=true` during incidents.
